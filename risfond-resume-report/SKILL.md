# 锐仕方达猎头简历报告制作规范

## 触发条件（必须加载本skill）

当用户要求生成简历报告时，**必须首先加载本skill**，不得跳过：
- 用户说"生成报告"、"写简历报告"、"出报告"
- 用户提供候选人资料并要求生成docx
- 用户说"用skill生成"（明确提示应使用skill）
- 任何涉及候选人简历报告生成的请求

**⚠️ 绝对不要**：不加载skill直接生成报告。本skill包含 mandatory 确认步骤和格式规范，跳过会导致流程错误。

## 工作流程
- DeepSeek（当前模型）直接完成全流程：分析→撰写→生成.docx
- 仅当用户明确说"用多模型""启用delegate"等指令时才切换到多模型协作

### 批量生成模式
当用户一次提供多个候选人资料时：
- 一次性分析所有候选人，分别输出每人的核心优势草稿（合并到一条消息中）
- 用户确认后，写一个Python脚本批量生成所有报告（而非逐个生成）
- 脚本中用函数封装每个候选人的生成逻辑，统一调用
- 最后一次性 Invoke-Item 打开所有文件
- 公司介绍搜索也批量并行执行，避免串行等待

### 流程（默认模式）
1. **DeepSeek分析**：分析候选人资料，规划报告内容结构
2. **输出【核心优势】草稿**：在对话框展示给用户确认，用户说OK后再继续
   - **⚠️ 必须等待用户确认后才能生成报告，绝对不能跳过此步骤**
   - 草稿只输出核心优势文本，不包含工作经历/薪资等
   - 用户通常会迭代2-4轮修改（调整顺序、措辞、增删条目）
   - 核心优势确认后，用户可能继续补充信息（年龄、到岗周期等），记下来一并写入报告
   - **常见修改方向**：第1条只留学历、第2条将设计能力前置编程后放、去除非岗位相关描述
3. **确认后撰写**：通过 `delegate_task` 委托 doubao 模型撰写完整报告（若 delegation 未启用则由 DeepSeek 直接执行生成脚本）
   - 首选：doubao-seed-2-1-pro-260628（volcengine-ark）
   - ⚠️ 若 API 429 限流（Safe Experience Mode 触发），降级至 doubao-seed-2-0-pro-260215
   - 子代理提示词模板要点：告知完整资料、要求 python-docx 生成、中文宋体+西文Times New Roman五号10.5pt、行距1.5倍、基本信息4行×2列无边框表格、工作经历/项目经历标题行#2E75B5、汇报对象/下属人数#000000、核心优势自然段落不拆两层、销售岗默认带项目经历板块（项目介绍/项目职责/项目业绩三段）、顾问名Alfred、薪资格式Xk*X薪=XXw税前、必须用qn('w:eastAsia')设置中文字体为宋体否则Word渲染为MS明朝
4. **验证与交付**：检查内容完整性，Invoke-Item 自动打开文件

### 用户补充信息处理
用户常在提供候选人资料时附带简短笔记（如"前期商业调查 收购 商业谈判"、"和正 土建尾声 设备进场安装阶段进入"、"看机会原因：资金链断裂"等），这些是**权威信息**，必须：
- 直接写入报告对应模块（职业状态/看机会原因/工作职责等）
- 不要质疑或改写用户提供的原始表述
- 如果笔记中包含与简历原文矛盾的信息，以用户笔记为准

### 格式修改的纪律（用户偏好）
- **用户提出任何格式/排版/样式修改时，主动询问**："这是单次要求还是长期规范？"
- 如果用户确认是长期规范 → 保存到 memory + 更新本 skill
- 如果用户说"梳理一下要求形成记忆" → 立即将要求保存到 memory 并更新 skill
- 如果用户未明确 → 默认按单次处理，只修改当前报告
- 用户讨厌冗余重复回答，要求简洁。格式更新后只确认"已完成"+关键结果
- 用户讨厌冗余重复回答，要求简洁。格式更新后只确认"已完成"+关键结果

### 参考文件的使用原则
- **整体以本 skill 为主要框架**，参考文件（用户提供的旧报告 .doc/.docx）仅用于修订 skill 中不吻合的细节
- 当 skill 规范与参考文件格式冲突时，以参考文件为准修订 skill
- 参考文件仅用于校准版式样式，不直接套用其内容结构
- 分析参考文件格式时，需用 Word COM 转换为 .docx 后用 python-docx 读取，检查 bold、size、color、alignment、space_before 等属性
- 分析参考文件时需检查图片：用 `zipfile` 提取 `word/media/` 下的图片，用 `PIL` 查看尺寸，判断哪些是标题图片、哪些是装饰性分隔线

### "测试效果"场景
当用户说"写一份简历报告测试下效果"时：
- 快速生成，不要过多讨论
- 先出核心优势草稿确认，用户OK后立即生成docx
- 用户会在生成后给出修改反馈，按反馈迭代

## 回复纪律

### 流程纪律
- **⚠️ 必须等待用户确认核心优势后才能生成报告**
- 绝对不能跳过确认步骤直接生成docx
- 用户说"生成"、"OK"、"确认"等肯定词后才继续

### 简洁性纪律
- **不要重复已经保存的内容**
- 创建/更新操作后，只确认"已完成"+ 关键结果
- 让用户自己查看skill获取完整信息
- 避免在回复中复述skill内容

### 跨平台分享
- 用户希望skill可跨平台分享（Hermes/Coze/Dify等）
- 保持skill内容独立完整，不依赖Hermes特定功能
- 使用通用markdown格式，便于其他平台读取

## 一、整体排版结构

### 1. 标题信息（页眉区）
推荐职位：XXX	推荐顾问：Alfred
推荐日期：YYYY-MM-DD（或 YYYY-M-D 均可，如 2026-3-3）
⚠️ 推荐职位行可选——如果用户未提供目标职位，可省略此行直接从基本信息开始

⚠️ **图片使用规则**：
- **不要锐仕方达企业logo图片**
- **不要装饰性分隔线图片**（含页眉横幅长条图，如 1180×64 顶部横幅）
- **默认行为①**：如果用户提供了参考文件（.doc/.docx），提取参考文件中的模块标题图片（基本信息/教育背景/工作经历/项目经历，特征是 580×48 左右的狭长横幅图）作为对应模块的标题
- **默认行为②**：无参考文件时，使用 skill `assets/` 目录内置的 4 张标准标题图片（title_basic_info.png / title_education.png / title_work_experience.png / title_project_experience.png）
- 标题图片替代的是【】文字标题的位置，不是额外添加

### 2. 候选人基本信息卡片（4行×2列无边框表格）
使用**无边框表格**布局（不用逐行段落），左列右列对齐：
| 姓名：XXX | 性别：男/女 |
| 年龄：XX岁 | 学历：本科/硕士/大专 |
| 籍贯：省份-城市 | 婚姻：已婚/已婚已育/未婚 |
| 现居：省份-城市 | （留空） |

- ⚠️ **没有"到岗周期"行**——到岗周期并入【职业状态】（如"离职，1个月内可尽快到岗"）
- 表格无边框、无底色，单元格内文字宋体五号不加粗
- 用户没提供某字段时对应单元格留空

### 3. 综合评估（核心卖点区）
严格按顺序，固定6-7个模块（面试安排可选）：
- **【职业经历】**（不加粗），时间倒序，格式：**YYYY/MM-YYYY/MM (X年X个月) 公司 职位**
  - ✅ 正确：`2008.12-2026.06 (17年6个月) 诺基亚 山东区域销售负责人`
  - ❌ 错误：`2008.12-2026.06 诺基亚 山东区域销售负责人 (17年6个月)`
  - 时长必须紧跟时间，不要放在职位后面
- **【核心优势】**（不加粗），见"三、写作要点"
- **【职业状态】**（不加粗）在职/离职 + 到岗周期，合并一句（如"离职，1个月内可尽快到岗。"）。离职可补充原因说明
- **【家庭情况】**（不加粗）婚姻、家庭位置、通勤。如"已婚已育，家在XX，通勤XX分钟"
- **【薪资结构】**（不加粗）目前薪资 + 期望薪资。格式见下方
- **【意向度】**（不加粗）高/中/低
- **【面试安排】**（不加粗）提前沟通（可选，部分报告省略）
- ⚠️ **所有【】标签默认不加粗**

### 4. 教育背景
**格式要求：**
- **标题**：如有参考文件图片则用图片，无则用【教育背景】文字（不加粗）
- 使用**五号字号（10.5pt）**，行间距1.5倍
- **无段前距，无开头空格**
- 每行一段，格式：YYYY/MM-YYYY/MM  学校  专业  学历(统招/非统招)
- 多学历按时间倒序排列

**示例：**
```
【教育背景】
2016/01-2018/12  中国人民大学  金融科技MBA  硕士
1999/09-2002/07  大连交通大学  信息工程  硕士
1995/09-1999/07  大连交通大学  计算机科学与工程  本科
```

### 5. 工作经历（每段）
**标题**：如有参考文件图片则用图片，无则用【工作经历】文字（不加粗）
标题行（蓝色加粗#2E75B5）：YYYY/MM-YYYY/MM(X年X个月)	公司	职位
公司介绍：50-100字，包含成立时间、总部/上市信息、主营业务、核心产品/客户、规模数据。优先通过搜索获取，避免只写一句话。**公司介绍内容不加粗**。
Base地点：（可选）
汇报对象：（标签加粗+#000000）
下属人数：（标签加粗+#000000，可选，0人也要写"下属人数：0"）
工作职责：（加粗） 1、 2、...
工作业绩：（加粗，如有） 1、 2、...
**离职原因/看机会原因：**XXX（写在工作职责最后一行，标签加粗）

**判断规则：**
- 在职人员（最近一段工作经历标注"至今"）→ 写"**看机会原因：**"
- 离职人员（所有工作经历都已结束）→ 写"**离职原因：**"
- 只有最近一段工作经历需要写原因，之前的工作经历不写

**同公司多岗位合并规则：**
- 同一公司多段经历合并为一条工作履历，时间取最早到最晚
- 职位栏写"职位A/职位B"（如"电气工程师/设备电气主管"）
- 职业经历列表中只列一条，但在工作经历正文中按时间段分别描述各岗位职责
- 职业经历列表中的职位写"电气工程师/设备电气主管"，不拆成两条

### 6. 项目经历（独立板块，销售/项目型岗位默认需要）
候选人操盘的重大项目单独成板块，位于**工作经历之后、尾部声明之前**。

**标题**：如有参考文件图片则用图片，无则用 skill assets 的 `title_project_experience.png`，再无则用【项目经历】文字（不加粗）

**每段项目结构**：
- 标题行（**蓝色加粗 #2E75B5**，与工作经历标题行同款）：`YYYY/MM-YYYY/MM	项目名称	项目角色`
  - 项目角色 = 候选人在项目中承担的职责定位（如"销售及项目管理"、"销售总监"、"项目总负责人"）
- `项目介绍：`（标签加粗，内容不加粗）1-3 句项目背景/规模/目标
- `项目职责：`（标签加粗，内容不加粗）编号列出 1、2、...
- `项目业绩：`（标签加粗，内容不加粗）编号列出，必须量化（中标金额/签约额/标杆意义）

**示例：**
```
2024/04-2026/07	国内某知名智算服务商智算项目	销售及项目管理
项目介绍：国内某知名智算服务商智算中心建设项目，竞争智算项目建设硬件标的。
项目职责：1、全面管理项目从立项到交付的全链条工作；2、击败竞争对手，高价中标，定义项目方向和后续目标。
项目业绩：1、公司首个智算中心推理集群建设，树立行业标杆；2、完成几千万级别订单，开拓新业务方向。
```

**判断规则：**
- 项目按**时间倒序**排列，时间段可与工作经历重叠（项目是工作经历中的亮点提炼，不冲突）
- 项目选择标准：有量化业绩、标杆客户/行业首创意义的项目；销售岗可列 5-10 个，技术/工程岗 2-4 个
- 客户保密需要时项目名用"国内某知名XX公司XX项目"（如"国内某知名智算服务商智算项目"）
- 项目介绍/项目职责/项目业绩三个子段齐全；确实无业绩数据时可省略"项目业绩："段，但不能三段都省
- 销售岗的项目经历是业绩的核心证据，**默认需要**；纯技术岗候选人简历无项目信息时可省略整个板块

### 7. 尾部声明
以上是我公司对候选人XXX的调研分析报告，文件中有需要保密的内容，请传阅时严格控制在与此次招聘有关的负责人手中。此文件的内容是我公司与候选人之间的面试总结。请贵公司根据人才情况和公司实际情况给予其相应待遇。
调研单位：锐仕方达人才科技集团有限公司常州第一分公司

## 二、表达习惯

- 第三人称"人选"，正式客观
- 时间格式：YYYY/MM-YYYY/MM (X年X个月)，在职标"至今"
- 动词开头：负责、主导、建立、优化、提升、统筹、完善、推行、搭建
- 数据量化：管理X人、销售额X万、成本降低X%、退货率降低至X%
- 薪资格式：XXk*X薪=XXw税前  综合年薪：XXw税前
- **非标准薪资结构处理**：当候选人薪资不是标准"月薪×月数"结构时（如base+绩效/奖金、base+季度奖、固定月薪+半年度/年度绩效等），按实际结构写明：
  - ✅ 示例：`base 18k*12+半年度绩效（64开）=36w税前`
  - ✅ 示例：`固定月薪25k+季度绩效（平均3个月）=38w税前`
  - ✅ 示例：`base 20k*13+年度奖金（2-4个月）=32w税前`
  - 综合年薪仍按"=XXw税前"结尾，保持统一格式
- 顾问名固定 Alfred
- **离职原因/看机会原因区分**：
  - **已离职人员**：写"**离职原因：**"（加粗标签，内容不加粗）
  - **在职人员**：写"**看机会原因：**"（加粗标签，内容不加粗）
- 离职原因要具体说明，如：
  - "公司业务萎缩，企业目前被收购"
  - "寻求更有前景更大的平台发展"
  - "技术路线混乱不清晰，方向不明确"
  - "公司高层人员变动较大，影响工作开展"
  - "家族式企业，晋升空间小"
- 到岗周期表述：1周内/X周/2周/X个月/1个月可尽快

### 离职原因中性化表述指南

候选人提供的原始离职原因可能包含负面信息（业绩不达标、被裁员、与领导不和、丢大单等），直接写入报告会影响客户对候选人的评价。必须将其**弱化个人失误、归因于公司层面调整或客观因素**。

**核心技巧：**
- ❌ 避免说：KPI重、业绩不达标、丢单、被裁员、与领导不和、能力不够
- ✅ 改为说：考核体系调整、组织架构优化、业务线调整、寻求更匹配的平台、公司战略方向变化
- ✅ 把焦点从"过去为什么不行"转移到"未来为什么更适合"

**常见敏感原因的中性化模板：**

| 原始原因 | 中性化表述 |
|---------|-----------|
| KPI太重/业绩压力大 | 公司近期调整了销售团队的考核体系与业绩目标设定，人选综合评估后决定寻求与自身资源和能力更匹配的平台发展 |
| 丢了大单/业绩不达标 | 公司近期进行组织架构优化与业务线调整，部分销售团队发生变动，人选希望寻求更稳定的发展平台 |
| 被裁员/部门解散 | 公司进行组织架构优化与业务线调整，人选希望寻找更能发挥自身资源优势的机会 |
| 与直属领导不和 | 公司高层人员变动较大，影响工作开展，人选希望寻求管理风格更匹配的平台 |
| 工资低/涨薪慢 | 人选在该公司已服务X年，近期公司战略方向发生调整，人选希望寻找更能发挥自身优势的机会 |
| 加班太多/工作强度大 | 公司近期业务节奏发生调整，人选希望寻求工作强度与生活质量更平衡的平台 |

**推荐版本结构（三选一）：**
- **归因公司调整（最常用）**："公司近期调整了XX，人选综合评估后决定寻求与自身资源和能力更匹配的平台发展"
- **归因组织架构变动**："公司近期进行组织架构优化与业务线调整，人选希望寻求更稳定的发展平台"
- **强调主动选择**："人选在该公司已服务X年，近期公司战略方向发生调整，人选希望寻找更能发挥自身优势的机会"

**当用户主动要求优化离职原因时：**
- 先展示原始原因，然后给出2-3个中性化版本供用户选择
- 让用户选定后再写入报告
- 不要在未经用户确认的情况下直接改写

## 三、核心优势写作要点

### 内容结构
每条优势是一个完整的自然段落，用数字编号（1、2、3、4）。

### 标准写法（4条结构）

从8份报告总结的标准模式：

**第1条：学历背景（只写学校+专业，不写年限）**
- 格式：`人选本科/硕士毕业于XX大学（985/211/QS排名）XX专业`
- 多学历合并写：`人选本科硕士毕业于XX大学XX专业`
- ✅ 示例：`1、人选本科毕业于西北工业大学（985 211）电气工程及其自动化专业。`
- ✅ 示例：`1、人选本科硕士毕业于南京理工大学（211）材料科学与工程专业。`
- ✅ 示例：`1、人选本科毕业于合肥工业大学（211）微电子专业，硕士毕业于加拿大西安大略大学（QS120）通信与信号处理专业。`
- ✅ 示例：`1、人选本科毕业于河北工程大学科信学院电气工程及其自动化专业。`
- ⚠️ **只写学校+专业，不附带工作年限、技能等**
- ⚠️ 非统本/专科 → 不在核心优势中写学历

**第2条：工作年限 + 行业经验 + 核心专业能力（最重要的一条）**
- 结构：`人选拥有X年XX经验，具备XX能力（设计/制图类前置），熟悉EPLAN/CAD/XX，擅长XX（编程/调试），能够独立完成从XX到XX的全流程`
- **设计/电气制图类能力置于编程之前**（如 EPLAN → AutoCAD → PLC → HMI → 调试）
- 内容最丰富，突出行业背景、专业深度、量化成果
- ✅ 示例：`2、人选拥有7年电气工程设计研发经验，具备起升、回转、变幅等工程机械电气系统设计能力，熟悉EPLAN电气设计，擅长欧姆龙、汇川等品牌PLC编程及威纶通、Proface等HMI开发，能够独立完成从电气方案设计到调试运行的全流程工作。`
- ✅ 示例：`2、人选拥有14年汽车质量管理经验，熟悉汽车零部件研发生产，负责全模块质量管理，涉及研发生产、供应商、客户质量管理。熟悉铸铁铸钢工艺，机械加工工艺...精通APQP、FMEA、PPAP、MSA、SPC、8D、QC七大手法等核心质量工具。`
- ✅ 示例：`2、人选拥有17年电气工程领域深厚经验，拥有汽车行业、智能仓储物流等行业经验，系统掌握电气设备规划、设计、安装、程序编写与调试全流程能力，人选精通西门子、倍福、CodeSys等PLC编程，熟悉EPLAN电气设计与WINCC/Intouch上位机开发...`

**第3条：技术工具/体系认证/专业技能**
- 聚焦具体工具、体系、标准、方法论
- ✅ 示例：`3、熟悉ISO9000/14000/13485/22000/45001/IATF16949/VDA/OHSAS18000/SA8000/EICC/BSCI/SEDEX等相关体系及品质控制手段（6sigma、精益生产、五大工具等），有超过十年的培训导师及推进经验。`
- ✅ 示例：`3、熟练使用Matlab/Simulink进行系统建模、传递函数设计，精准计算电流环带宽与跟随性能...熟悉雷击、浪涌、群脉冲等可靠性实验，三米法电磁兼容试验...`

**第4条（可选）：语言能力/团队管理经验/其他突出优势**
- 外语能力突出时单独成条
- 团队管理经验突出时单独成条
- ✅ 示例：`4、人选英语熟练可以无障碍沟通，具备英文文献材料的吸收能力。`
- ✅ 示例：`4、人选拥有百人规模质量团队管理经验。`
- ✅ 示例：`4、人选拥有4门外语能力：英语：精通...柬埔寨语：日常沟通...西班牙语：日常沟通...越南语：日常沟通`

### 写作风格要点

1. **语言精炼**：每条优势1个完整段落，不拆分小标题
2. **量化数据**：X年经验、X万销售额、X人团队、X%提升
3. **行业关键词**：突出目标岗位所在行业的专业术语
4. **工具/体系/标准**：具体列出掌握的工具、体系、认证
5. **逻辑递进**：学历→经验→技能→附加优势

### ⚠️ 注意事项
- 核心优势不加粗，使用默认黑色字体
- 推荐写3-4条，信息密度高才写4条
- 只能从候选人实际工作经历中提取，不添加无关信息
- 用户没提供年龄字段时，基本信息中直接省略

### 用户偏好风格（高信息密度版）

部分用户偏好更精炼的核心优势，特点：
- **0-1经验是核心亮点**：必须突出"从0-1工厂建设"经历，这是关键卖点
- **产值/规模数据前置**：年营收X亿、项目规模X万吨等量化数据非常重要
- **年营收/产值不能跨公司合计**：年营收、产值、产能等反映平台规模的指标不是个人销售业绩，不能多段经历加总。应在每段工作经历中分别标注该公司年营收。❌ 错误示例："主导项目年营收合计超28亿（迪牧8亿、汇邦20亿）"。✅ 正确做法：在山东和正的公司介绍中写"年营收X亿"，在临沂汇邦的公司介绍中写"年营收20亿"
- **区分"个人业绩"与"公司规模"**：销售额/签约额/降本金额等个人可直接归因的业绩可以合计；年营收/产值/产能等反映平台规模的指标只能分段标注
- **不提证书**，聚焦实战经验
- **项目成果前置**，用具体项目名称和规模证明能力
- **语言直接**，"18年加氢工艺生产管理经验"而非"人选拥有18年化工行业经验"

示例（3条结构，不含学历因为非统招）：
```
1、人选拥有18年加氢工艺生产管理经验，具备3段从0-1工厂建设经历，熟悉粗苯加氢、混合苯加氢、煤焦油加氢、生物柴油（HVO）、生物航煤（SAF）、废矿物油加氢精制等加氢工艺全流程建设。
2、精通制氢、加氢精制、加氢裂化、加氢异构、萃取蒸馏、常减压等工艺，熟悉加制氢装置关键设备结构及运行维护，掌握酸性水汽提、硫磺回收等配套装置操作要点。
3、主导25万吨/年废矿物油加氢精制项目（国内单套最大废油再生Ⅱ类润滑油基础油装置）和20万吨/年生物柴油（HVO）/生物航煤（SAF）项目建设投产，推动多项技改落地实现产能及质量提升。
```
（年营收数据在工作经历的公司介绍中分别标注：临沂汇邦-20亿、唐山迪牧-8亿）

**学历表达规则：**
- ✅ 统招本科/硕士 → 写进核心优势第1条
- ❌ 非统招本科/专科 → 不写进核心优势，只在教育背景中体现
- ⚠️ **判断方法**：通过教育经历时间线判断是否统招
  - 如果大专毕业后再读本科（如大专2008年毕业，本科2010年入学），说明是专升本/非统招
  - 如果本科和大专时间连续（如大专2008年毕业，本科同年入学），可能是统招专升本，需确认
  - 如果只有本科一段经历且无大专，通常是统招
  - 简历上写"本科"不代表统招，必须看时间线

**0-1经验表达模板：**
- "具备X段从0-1工厂建设经历"
- "从0-1搭建团队，主导工艺路线更换、老厂房改造"
- "从0-1主导工厂建设，年营收X亿"

### 销售类岗位核心优势模板（B2B大客户销售/企业客户经理）

销售岗位核心优势与技术/工程岗不同，重点不在工具/证书/体系认证，而在**业绩数据、客户层级、解决方案能力、生态资源**：

**4条结构：**
- 第1条：学历（统招本科写，非统招/专科省略，同标准规则）
- 第2条：X年XX行业B2B大客户销售经验 + 细分领域（云计算/AI算力/工业软件/SaaS/供应链等）+ 擅长对接的决策层级别（CIO/CXO/厂长/副总/董事长）+ 解决方案类型
- 第3条：业绩量化数据前置——年度销售额/累计项目金额 + 标杆项目（项目名+金额+关键产品/方案）+ 客户类型（国企/上市民企/外资/头部客户）
  - **变体：「相关业绩」子结构**（业绩横跨多家公司、数据丰富时用，更醒目）：
    ```
    3、相关业绩：
    3.1 公司A-任职期间年均销售额7000万、单笔订单2000-3000万，促成阿里存储项目签约1亿+、整体框架5亿+；
    3.2 公司B-期间操盘阿里云GPU/存储项目年中标额超5亿；
    3.3 公司C-期间落地讯飞AI存储项目；
    累计操盘项目金额超20亿，客户覆盖阿里、腾讯、科大讯飞等头部企业。
    ```
  - 格式要点：公司名在前用"-"连接业绩；每条以分号结尾；最后一行"累计..."汇总总金额+头部客户名单；跨公司业绩归属必须清晰，不得混淆
- 第4条：生态资源（原厂合作关系如阿里/西门子/IBM/华为/AWS等）+ 渠道管理能力 + 团队管理经验 + 语言/其他突出优势

**制造/工业软件销售示例：**
```
1、人选本科毕业于三峡大学机电一体化专业，具备机电一体化技术基底，可快速对接设备、工艺、IT技术团队，并将技术优势转化为高管认可的战略价值与沟通话术。
2、人选拥有近16年大客户销售经验，10年深耕制造领域全周期顶层经营，精通供应链S&OP/APS、数字化PLM/MES/QMS工业软件、工业大型设备及软硬一体化解决方案销售，熟悉大型制造企业高层决策逻辑，擅长对接工厂总经理、集团供应链总监、分管副总等决策层，独立操盘多层级复杂招投标项目。
3、人选持续稳定交付年度千万级业绩：LG工业设备区域峰值年销3000万+、科瑞恩西门子数字化软硬一体项目年度新增1200万+、悠桦林供应链APS软件年均900万+，累计操盘多个百万级标杆项目，擅长全链路项目闭环管控及风险前置管理。
4、人选长期维护阿里、IBM、西门子原厂及行业系统集成商高层关系，擅长全域生态资源统筹与多方利益平衡，可联动原厂、集成商输出一体化解决方案，具备从0到1开拓细分赛道头部客户的经验。
```

**云计算/AI算力销售示例：**
```
1、人选拥有6年云计算和AI领域销售经验，深耕阿里云生态体系，熟悉阿里云产品、数云融合解决方案及公共云到智算产品的全栈销售，累计主导大型项目18+，个人项目成交金额超5600万。
2、人选具备AI大模型及GPU算力领域深度产品知识，熟悉机器学习平台架构，能够针对大模型训练、推理及大数据场景制定计算、存储、网络资源方案并主导POC测试，成功交付世界模型训练（443万，12台A100）、医疗大模型微调（90万）、推理资源4090（228万）等标杆项目。
3、人选拥有丰富的阿里云渠道及直销经验，曾任职阿里云核心代理商青莲网络，个人项目金额3074万+，月均业绩80万+，后期管理4人团队；后于神州数码负责阿里云产品市场拓展及数云融合方案推广，项目总金额2000万+，擅长对接CIO、CXO等高层决策人。
```

**注意事项：**
- 业绩数据必须量化（金额、台数、项目数），不能写"业绩优秀""多次超额完成"
- 跨公司业绩列举时必须明确标注每笔业绩对应的任职公司，不得混淆不同公司的业绩归属
- 业绩描述需精简，避免"行业标杆""业务样板""树立典型案例"等无实质数据的描述性文字，核心保留「项目名称+金额+核心客户」即可
- 区分个人业绩与公司规模（同标准规则：年营收等平台指标分段标注，销售额等个人业绩可合计）
- 突出产品名称（如阿里云、AWS、S&OP、APS、A100/H100 GPU、rafay平台等），体现产品专业度
- 决策层对接能力要写具体级别（CIO/厂长/副总/董事长），不能只写"高层"
- 不提CET/计算机二级等基础证书，聚焦实战销售经验与业绩成果

## 四、排版与字体格式规范（关键）

### 字体规范（中西文分离）
- **中文字体**：宋体 (SimSun)（通过 `qn('w:eastAsia')` 设置）
- **西文/数字字体**：Times New Roman（`font.name` 设置）——与原版 .doc 一致，数字和英文渲染为 Times New Roman
- **全文字号**：五号（10.5pt）
- **标题字号**：五号（10.5pt）
- **工作经历/项目经历标题行**：五号，加粗，蓝色（#2E75B5）
- ⚠️ **必须用 `qn('w:eastAsia')` 设置中文字体为宋体**，否则 Word 会渲染为 MS 明朝

### 字体颜色（从原版 .doc 反编译）
| 内容 | 加粗 | 颜色 |
|------|------|------|
| 推荐职位/推荐日期 | Bold | 默认（黑色） |
| 综合评估： | Bold | 默认（黑色） |
| 【职业经历】【核心优势】【职业状态】【家庭情况】【薪资结构】【意向度】【面试安排】【教育背景】【工作经历】等 `【】` 段 | 不加粗 | 默认（黑色） |
| 核心优势编号内容（1、2、3...） | 不加粗 | 默认（黑色） |
| 职业状态/家庭情况/薪资结构/意向度的内容 | 不加粗 | 默认（黑色） |
| **工作经历标题行（时间+公司+职位）** | **Bold** | **#2E75B5**（全文仅标题行用彩色） |
| 公司介绍：标签 | **Bold** | 默认（黑色）；介绍内容不加粗 |
| 汇报对象： | **Bold** | **#000000** |
| **下属人数：** | **Bold** | **#000000** |
| 工作职责：/工作业绩： | Bold | 默认（黑色） |
| 职责/业绩编号内容 | 不加粗 | 默认（黑色） |
| 项目经历标题行（时间+项目名+角色） | **Bold** | **#2E75B5**（与工作经历标题行同款） |
| 项目介绍：/项目职责：/项目业绩：标签 | Bold | 默认（黑色）；内容不加粗 |
| 离职原因/看机会原因标签 | Bold | 默认（黑色） |

| 教育背景行 | 不加粗 | 默认（黑色） |
| 尾部声明 | 不加粗 | 默认（黑色） |

⚠️ 每个 run 必须显式设置 `run.font.color.rgb` 为 BLACK 或 BLUE，否则渲染偏浅。

### 段落格式
- 页边距：上下2.54cm，左右3.17cm
- 行距：1.5倍行距
- 段前段后：0pt
- 工作经历标题行内三段用 `\t` 分隔（时间\t公司\t职位）
- 每段工作经历之间用空行分隔
- **基本信息卡片中，每个字段独占一行，字段之间不要空行**
- **教育背景内容无段前距，无开头空格**

```python
style.paragraph_format.line_spacing = 1.5
style.paragraph_format.space_before = Pt(0)
style.paragraph_format.space_after = Pt(0)
# 设置中文字体（East Asian）
style.element.rPr.rFonts.set(qn('w:eastAsia'), '宋体')

BLUE = RGBColor(0x2E, 0x75, 0xB5)
BLACK = RGBColor(0, 0, 0)

def R(p, text, bold=False, size=10.5, color=None):
    run = p.add_run(text)
    run.bold = bold
    run.font.size = Pt(size)
    run.font.name = 'Times New Roman'   # 西文/数字
    run.font.color.rgb = color if color else BLACK
    # 设置中文字体（East Asian）
    run.element.rPr.rFonts.set(qn('w:eastAsia'), '宋体')
    return run

def add_p(text='', bold=False, size=10.5, color=None):
    p = doc.add_paragraph()
    if text:
        R(p, text, bold=bold, size=size, color=color)
    return p

def add_basic_info_table(info):
    """基本信息 4行×2列 无边框表格。info: [(左列, 右列), ...]，如
    [('姓名：张海默', '性别：男'), ('年龄：46岁', '学历：本科'),
     ('籍贯：江苏-苏州', '婚姻：已婚已育'), ('现居：上海', '')]"""
    tbl = doc.add_table(rows=len(info), cols=2)   # 默认样式无边框
    for i, (left, right) in enumerate(info):
        for j, text in enumerate((left, right)):
            cell = tbl.cell(i, j)
            p = cell.paragraphs[0]
            p.paragraph_format.line_spacing = 1.5
            if text:
                R(p, text)
    return tbl

def add_project(time_range, proj_name, role, intro=None, duties=None, results=None):
    """项目经历段落：标题行蓝色加粗，标签加粗内容不加粗"""
    p = doc.add_paragraph()
    R(p, f'{time_range}\t{proj_name}\t{role}', bold=True, color=BLUE)
    if intro:
        p = doc.add_paragraph(); R(p, '项目介绍：', bold=True); R(p, intro)
    if duties:
        for k, d in enumerate(duties):
            p = doc.add_paragraph()
            if k == 0:
                R(p, '项目职责：', bold=True)
            R(p, d)
    if results:
        for k, d in enumerate(results):
            p = doc.add_paragraph()
            if k == 0:
                R(p, '项目业绩：', bold=True)
            R(p, d)
```

## 七、常见陷阱

### 7.1 文档锁定问题
生成 .docx 后 Word 会锁定文件。如需重新生成覆盖，必须先关闭 Word 窗口或杀掉 WINWORD 进程。
```powershell
Get-Process WINWORD -ErrorAction SilentlyContinue | Stop-Process -Force
```
或换用新文件名保存避免冲突。

### 7.2 旧版 .doc 文件分析
python-docx 读不了旧版 .doc 格式。需用 Word COM 转 .docx 后再分析。参见 `references/candidate-file-reading.md`。

### 7.3 hermes CLI 路径空格问题
Windows 下 hermes.bat 路径含空格，不能直接用 `cmd /c` 或 `hermes.bat`。正确方式：
```powershell
& "D:\Hermes Agent CN Desktop\data\desktop-bin\hermes.bat" tools enable <name>
```

### 7.4 python-docx 中文字体必须设置 East Asian
`font.name = 'Times New Roman'` 只设置西文/数字字体，中文字体必须通过 XML 操作：
```python
from docx.oxml.ns import qn
style.element.rPr.rFonts.set(qn('w:eastAsia'), '宋体')
run.element.rPr.rFonts.set(qn('w:eastAsia'), '宋体')
```
不设置的话 Word 会渲染为 MS 明朝而非宋体。西文不要设成'宋体'——原版报告西文/数字均为 Times New Roman。

### 7.5 公司介绍需要搜索补充（强制执行）
公司介绍必须50-100字，**即使参考文件中有公司介绍字段，如果内容为空或不足50字，也必须搜索补充**。常见情况：
- 参考文件中公司介绍字段存在但为空（如 `公司介绍：` 后面没有内容）
- 参考文件中公司介绍只有一句话
- 用户只提供了简历PDF，没有参考报告

**搜索内容**：成立时间、总部/上市信息、主营业务、核心产品/客户、规模数据（注册资本/占地面积/年营收等）。
**搜索方式**：使用 `china-web-search` skill 中的 search.py 脚本搜索公司全称。
**不可接受的情况**：直接写 `add_p('公司介绍：', bold=False)` 而不提供具体内容。

**搜索失败时的降级策略**：
当Tavily/网络搜索不可用、返回空结果**或返回无关结果**时，不要留空或只写一句话。注意：Tavily对小微型私企经常返回完全不相关的公司（如搜"丹阳三冠新材料"返回其他上市公司年报），此时等同于搜索失败，不要使用无关结果。基于候选人简历原文提炼：
- 从简历中的"公司介绍"或"工作概述"段落提取关键信息
- 补充行业定位、主营业务、核心产品/客户等要素
- 如简历中信息极少，可用常识性描述（如"阿里云核心代理商，专注于阿里云产品销售"）
- 底线：即使搜索结果为空或不相关，公司介绍也必须达到50字左右，不可省略

### 7.6 同公司多岗位需合并履历
候选人在同一公司担任多个岗位时，工作履历合并为一条（时间取最早到最晚），职位栏写"职位A/职位B"。职业经历列表中只列一条。但在工作经历正文中，按时间段分别描述各岗位职责和业绩。

### 7.7 工作年限必须精确到月，禁止小数近似
- ❌ 错误：约17.5年、约4.5年、17.5年、4.5年
- ✅ 正确：17年6个月、4年8个月
- 计算方法：从入职月到离职月（或至今），精确计算年和月
- 职业经历列表和工作经历标题行都要用"X年X个月"格式
- 不要用"约"字，直接写精确数字

### 7.8 教育背景格式规范
- ❌ 错误：使用小六字号（9pt）、使用6.5pt、段前距过大、开头留空格
- ✅ 正确：使用**五号字号（10.5pt）**，行间距1.5倍，无段前距，无开头空格
- 五号 = 10.5pt（中文字号标准）
- 参考格式见第4节示例

### 7.9 【】标签格式
- 默认所有【】标签不加粗
- 包括：【职业经历】【核心优势】【职业状态】【家庭情况】【薪资结构】【意向度】【面试安排】【教育背景】【工作经历】

### 7.10 图片使用规则（自动执行）
- ❌ 错误：添加锐仕方达logo图片、添加装饰性分隔线图片（含页眉横幅长条图）
- ✅ 正确①：**如果用户提供了参考文件，默认自动提取参考文件中的模块标题图片（基本信息/教育背景/工作经历/项目经历，580×48 左右狭长横幅）作为对应模块的标题**
- ✅ 正确②：无参考文件时，默认使用 skill `assets/` 内置的 4 张标准标题图片（title_basic_info.png / title_education.png / title_work_experience.png / title_project_experience.png）
- ⚠️ **不要等用户提醒才提取图片**——提供参考文件时，第一步就应该用 Word COM + zipfile + PIL 提取并分析图片
- 标题图片替代的是【】文字标题的位置，不是额外添加
- 绝对不要添加企业logo或装饰性分隔线图片

### 7.11 必须包含工作经历
- 教育背景后必须接工作经历内容
- 如有参考文件图片则用图片作为标题，无则用【工作经历】文字标题（不加粗）

### 7.12 基本信息卡片不可遗漏
报告必须包含完整的候选人基本信息卡片（4行×2列无边框表格：姓名/性别、年龄/学历、籍贯/婚姻、现居），放在推荐日期之后、综合评估之前。到岗周期不单独列出，写入【职业状态】。

### 7.13 生成后必须验证内容完整性
- 生成 .docx 后，用 python-docx 读取并打印所有段落文本，确认【教育背景】【工作经历】等内容确实写入文件
- 常见遗漏原因：
  - Word 锁定文件导致 PermissionError，脚本实际未生成新文件 → 换用新文件名
  - 脚本逻辑错误导致段落未正确添加 → 检查 `add_p()` 调用和 `R()` 调用
  - 变量名重复覆盖（如 `p = add_p()` 后面又 `p = add_p()`）→ 用不同变量名
- 验证代码：
```python
from docx import Document
doc = Document(r'文件路径.docx')
for i, p in enumerate(doc.paragraphs):
    if p.text.strip():
        print(f'[{i}] {p.text[:60]}')
```

### 7.14 标题图片提取与使用
当用户要求"跟参考文件排版保持高度一致性"且参考文件包含标题图片时：
1. 用 Word COM 将 .doc 转为 .docx
2. 用 `zipfile` 提取 `word/media/` 下的所有图片
3. 用 `PIL.Image.open()` 查看尺寸，区分标题图片（580×48 左右狭长横幅）和装饰图/页眉横幅（如 1180×64、超大正方形 logo）
4. 将标题图片插入到对应模块位置（基本信息/教育背景/工作经历/项目经历）
5. **不要锐仕方达企业logo**、**不要装饰性分隔线/页眉横幅**
6. 图片替代的是【】文字标题的位置，不是额外添加
7. 参考文件缺某模块标题图时，用 skill `assets/` 内置标准图补齐

### 7.15 候选人信息时效性检查（关键）
当用户提供旧报告作为参考或候选人资料时，**必须主动检查信息的时效性**：
- 最近工作经历是否仍标注"至今"？实际可能已离职/入职新公司
- 职业状态是否仍写"在职"？实际可能已变动
- 公司是否已暴雷/被收购/更名/资金链断裂？（如"南都暴雷"、"资金链断裂"）
- 用户口头补充的信息（如"4月底南都暴雷我就离开了，5月初来的隆基"）**优先于参考文件的旧信息**

**处理流程**：
1. 读取参考文件后，向用户确认："参考文件显示候选人目前在XX公司，请问这是最新状态吗？"
2. 如果用户补充了新信息 → 以用户口述为准，在报告中更新最新经历
3. 如果参考文件已严重过时 → 不要直接按过时信息生成，等待用户确认
4. 离职原因/看机会原因以用户最新说明为准（如"认证工程师给我挂在另一个部门"）

**跳过确认步骤的情况**：
当用户**主动提供了明确的当前信息**（如具体薪资结构、离职原因、到岗时间、家庭情况等），且这些信息与参考文件不一致时，**直接使用用户提供的信息，无需再询问确认**。用户的显式输入本身就是最新的权威信息。例如：
- 用户说"目前13k*12薪"，参考文件写"11k*9+奖金" → 直接用13k*12薪
- 用户说"离职原因：板块小做配套" → 直接写入报告
- 用户说"单身，家在四川" → 直接写入家庭情况

**参考报告复用策略**：
当用户同时提供PDF简历和参考报告（.doc/.docx）时，参考报告的结构化内容（公司介绍、项目经历、工作职责描述等）可以直接复用，只需用用户提供的最新信息覆盖对应字段。这样可以节省公司介绍搜索时间，保持报告风格一致。

### 7.16 执行文件操作前必须列清单确认
用户明确要求：**执行文件移动/删除/整理等批量操作前，必须先列出具体操作清单，经用户确认后再执行最后一步。**
这是用户的工作纪律，必须遵守。

### 7.17 加粗文本必须用R()函数，禁止在add_p()中用markdown语法
- ❌ 错误：`add_p('**看机会原因：**内容')` → Word中会显示为 `**看机会原因：**内容`（星号原样输出）
- ✅ 正确：先 `p = add_p()`，再 `R(p, '看机会原因：', bold=True)` + `R(p, '内容')`
- python-docx 的 `add_run()` 不做 markdown 解析，`**text**` 不会被转换为加粗
- 所有需要加粗/变色/变字号的文本，必须通过 `R()` 函数的参数控制

### 7.18 Word COM 批量转换 .doc 文件的稳定性问题
同一 Word COM 会话中连续转换多个 .doc 文件时，第二个及后续文件容易出现 RPC 错误（0x800706BE / 0x800706BA）：
- **症状**：第一个 .doc 转换成功，后续文件报 "The remote procedure call failed" 或 "RPC server is unavailable"
- **原因**：Word COM 对象在连续操作后变得不稳定
- **解决方案**：每个 .doc 文件用独立的 Word COM 会话转换（每次 `New-Object -ComObject Word.Application` → 转换 → `Quit()`），而非一个会话中连续打开多个文件
- **备选方案**：如果独立会话仍失败，先 `Get-Process WINWORD | Stop-Process -Force`，等 3 秒后再重试
- **降级处理**：如果 .doc 始终无法转换，告知用户该文件无法读取，请用户提供 PDF/DOCX 格式或手动提供候选人信息
